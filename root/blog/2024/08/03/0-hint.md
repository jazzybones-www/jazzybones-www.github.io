# the jazzybones word: hint

a "hint" in computer science is basically a recommendation. for example, a
mobile app might work with both a horizontal and vertical layout, but look
better with one or the other. the app can "hint" to the operating system that it
should be displayed a certain way. the operating system could listen to the
hint, or it could just do its own thing. the idea is that ignoring the hint
won't be catastrophic, but it will lead to worse results.

this is a really bad name. like, if i'm doing a puzzle and i take a hint, that
implies that there is exactly one correct solution, that that solution is
actively being withheld from me, and that i need some help to figure out what
that solution is. with this definition, there are multiple correct solutions,
although one is considered the "best" one, and that solution is being directly
told to the program.

"hint" implies mystery. if i have to "take the hint", that implies that it's
possible to miss the hint, that the hint is obscure enough to be missed in the
first place. a hint in software is a direct message from one program to another
to do something in one specific way.

a better word might be "recommendation". the thing is that "hint" is so
ingrained in every software engineer's vocabulary, including mine, that we can't
change it now. if i were to write "recommendation" in a documentation,
everybody would go "what do you mean by that?" and i would have to clarify "i
mean hint". in fact, if some _other_ engineer used the word "recommendation",
_i_ would have the same reaction. at this point, engineering lingo has evolved
to use the word hint, and there's nothing that anybody can do about it.

it kind of feels like engineering jargon is slowly turning into its own
language. the only real effect this has that i can think of is that it makes
engineering harder to learn. you could argue that this isn't necessarily a bad
thing, if we used the word "recommendation" instead of "hint", it might make
some people think that they know what "recommendation" means when they really
don't. by clearly defining a "hint" as its own thing, we separate it from its
meaning outside of engineering.

this logic works for math (see: abelian groups, extension fields, etc.), but in
engineering so many things are based on other ideas that it's just impractical.

basically, engineers suck at naming things.
